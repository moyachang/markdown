# WFCIService 重構後 Spring Remoting 相容處理

## 【分享原因】
WFCIService 因應重構與架構升級，Spring Boot 由 2.5.12 升級至 3.5.6。其中 **Spring Remoting (RMI)** 因為安全性漏洞與跨語言限制，在 Spring 6 / Boot 3 中已遭正式移除。

然而，WFCIService 的 API 大量依賴 RMI 技術與後端 PASE 引擎通訊，且 **PASE Server 目前因現況無法同步升級 Spring Boot 版本**。為了讓新舊程式共存、確保功能不中斷，WFCIService 必須自行實作相容層。本次分享將著重於重構過程中對 RMI 底層機制的剖析與解決方案。

---

## 一、 何謂 RMI (Remote Method Invocation)?

**遠端方法呼叫 (Remote Method Invocation)** 是 Java 原生的分散式通訊 API。
* **用途**：允許在一台 JVM 上運行的程式，直接呼叫另一台遠端伺服器上的 Java 物件方法。
* **優點**：讓開發者在撰寫分散式應用程式時，不需要手動處理 Socket、序列化等底層網路通訊細節。

### 1. Spring Remoting RMI 呼叫流程
當 Client 呼叫遠端服務（例如 `userService.queryUser("A123")`）時，無法直接將程式碼丟到 Server 執行，必須經歷：**封裝呼叫資訊 -> 序列化傳送 -> Server 執行 -> 回傳結果**。

Spring 框架將傳統 RMI 的 `Stub` (客戶端代理) 與 `Skeleton` (伺服器端骨架) 進行了封裝，核心涉及三個關鍵類別：

```text
Client (呼叫端)
  │
  │ 執行：userService.queryUser("A123")
  ▼
RmiInvocationWrapper_Stub (客戶端代理物件)
  │
  │ 負責建立並序列化
  ▼
RemoteInvocation (遠端呼叫請求 DTO)
  │
  │ RMI 網路傳輸
  ▼
RmiInvocationHandler (伺服器端呼叫入口)
  │
  │ 解析並透過 Java Reflection 呼叫
  ▼
Real Service (真正的 PASE 服務)
  │
  ▼
回傳 Result
````

### 2\. 三個核心元件與職責

| 元件名稱 | 職責與角色 | 實務運作說明 |
| --- | --- | --- |
| **RemoteInvocation** | **封裝呼叫資訊** | 本質上是一個 `Serializable` 的 DTO（派工單），記錄了 Method Name、Parameter Types 與 Arguments。不負責執行任何邏輯。 |
| **RmiInvocationWrapper\_Stub** | **傳送呼叫請求** | 實作了 `RmiInvocationHandler` 介面，存在於 Client 端的代理物件（遙控器），負責將 RemoteInvocation 序列化後透過網路送到 Server。 |
| **RmiInvocationHandler** | **接收並執行呼叫** | Server 端公開的 RMI 遠端介面（公司櫃台）。收到 RemoteInvocation 後，解析其方法與參數，還原並執行真正的 Service，再將結果回傳。 |

> **💡 補充名詞：Stub (虛設常式)** 在台灣開發術語中，Stub 常指測試時寫死回傳值（Hard-code）的假功能（Mock 兄弟類）。但在 RMI 網路架構中，Stub 專指「客戶端代理存根」，負責處理底層 Socket 與通訊，讓本機程式誤以為遠端服務就在本機一樣。

二、 RMI 在 WFCIService 的工作情境
--------------------------

1.  **分散式通訊的橋樑**：WFCIService 本身不執行工作流引擎核心代碼，而是作為 RMI Client 連線到遠端的 PASE RMI Server。
2.  **透過 Spring 動態代理查找 (Lookup)**：在 `PASEConfig.java` 中，原先利用 Spring 的 `RmiProxyFactoryBean` 對接 RMI 註冊表（Port 1099），動態查找並包裝以下 PASE 核心遠端服務接口：
    *   `DBService`：資料庫與內部數據存取。
    *   `FileService`：遠端文件與附件管理。
    *   `FlowService`：核心流程、表單與 BPM 工作流引擎運作。
    *   `OrgService`：組織架構、角色與權限控制。
3.  **初始化 WFCI 執行實體**：建立服務提供者後，注入 `WFCIPool` 產生 `WFCI`（Workflow Client Interface）物件，供 Controller / Service 層使用。

### 【技術考古：為什麼 Ant 打包腳本有 `*_Stub.class` 但實體目錄找不到？】

在舊版 `build_dev.xml` 中可以看到 `<include name="pase/server/ServerServiceImpl_Stub.class" />` 等設定：

*   **以前**：Java 5 之前，必須使用 JDK 的 `rmic` (RMI Compiler) 工具對實作類別進行二次編譯，手動產出這些 Stub 實體檔案，否則 Client 無法連線。
*   **現在（後期）**：
    1.  Java 5+ 支援動態 Stub (Dynamic Proxy Stub)。
    2.  專案導入 Spring Remoting，`RmiServiceExporter` 會在執行期（Runtime）自動透過 Java 動態代理產生通訊代理。
*   **結論**：打包腳本中的 `*_Stub.class` 是沿用舊版本的**歷史遺留設定**，目前 Ant 執行時找不到會自動忽略 (Ignore)，實質上通訊早已交由 Spring 執行期動態代理處理。

三、 重構升版後的相容挑戰與方案選擇
------------------

### 1\. 為什麼不能直接使用 Spring Remoting？

Spring Boot 3.0 / Spring Framework 6.0 徹底移除了該功能，主要原因為：

*   **Java 反序列化漏洞**：惡意攻擊者可透過偽造 `RemoteInvocation` 讓伺服器在解包瞬間執行惡意程式碼。
*   **防火牆不友善**：RMI 使用隨機或特定 TCP 埠，不如標準 HTTP (80/443) 易於管理。
*   **跨語言能力差**：架構完全綁死在 Java 語言上。

### 2\. 方案評估

WFCIService 採用 **Strangler Pattern (絞殺者模式)** 進行漸進式遷移，舊的 API 必須繼續運行。

*   ❌ **方案 A：升級 PASE Server** -\> 現階段政治或技術因素無法動彈。
*   ❌ **方案 B：改用純 JDK RMI** -\> 協定不相容，因為 PASE Server 依然是 Spring Remoting 封裝。
*   ❌ **方案 C：自行重寫所有 Remoting 類別** -\> 工程量巨大，不切實際。
*   ✅ **方案 D：引入舊版隱含依賴 + 自建相容代理層** -\> **最終採用方案**。

四、 核心實作調整
---------

**關鍵成功因素**：直接引入 `spring-remoting-2.0.5.jar`，利用其原始的 `RemoteInvocation` 與 `RmiInvocationHandler` 類別，確保 Client 與 Server 的序列化協定完全一致。

### 1\. Maven 依賴配置

在父項目與 common 模組中引入舊版 RMI 依賴，提供反序列化所需的關鍵類別：

```
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-remoting</artifactId>
    <version>2.0.5</version>
</dependency>
```

### 2\. 自定義 RemoteServiceProvider 介面

擺脫對舊版 PASE jar 的直接依賴，改由 WFCIService 自行控制服務查找邏輯。

```
public interface RemoteServiceProvider {
    Object getService(String serviceName) throws Exception;
    String getRmiRegistryUrl();
}
```

### 3\. 自建相容層實作 (CompatibleRmiRemoteServiceProvider)

透過 **JDK Dynamic Proxy** 手動封裝 `RemoteInvocation`，藉此繞過已不存在的 `RmiProxyFactoryBean`。

```
@Override
public Object getService(String serviceName) throws RemoteLookupFailureException {
    // 1. 查詢快取
    Object cached = serviceCache.get(serviceName);
    if (cached != null) return cached;

    // 2. 取得對應的 PASE 介面 (例如 "DBService" → pe.pase.DBService)
    Class<?> serviceInterface = Class.forName(SERVICE_INTERFACE_MAP.get(serviceName));

    // 3. 從 RMI Registry 查找遠端物件 (此處會觸發 spring-remoting jar 的反序列化)
    Remote remoteObj = (Remote) registry.lookup(serviceName);
    RmiInvocationHandler handler = (RmiInvocationHandler) remoteObj;

    // 4. 建立動態代理，手動封裝舊版的 RMI 傳輸協定
    Object proxy = Proxy.newProxyInstance(
        serviceInterface.getClassLoader(),
        new Class<?>[] { serviceInterface },
        (proxyObj, method, args) -> {
            RemoteInvocation invocation = new RemoteInvocation(
                method.getName(), method.getParameterTypes(), args);
            return handler.invoke(invocation); // 透過舊版 Handler 呼叫遠端 PASE
        }
    );

    serviceCache.put(serviceName, proxy);
    return proxy;
}
```

### 4\. 配置調整 (PASEConfig.java)

*   **加入 `@Lazy`**：防止應用程式啟動時因遠端 PASE Server 未就緒而導致整台 WFCIService 掛點。
*   **增設 try-catch 容錯**：連線失敗時改為降級運行，而非阻斷啟動。
    
```
@Bean
@Lazy
public RemoteServiceProvider serviceProvider() {
    RemoteServiceProvider serviceProvider = null;
    try {
        logger.info("Initializing CompatibleRmiRemoteServiceProvider...");
        serviceProvider = new CompatibleRmiRemoteServiceProvider(
            sysConfig().getPaseIP(), sysConfig().getPasePort());
        sc.setAttribute(WFCIServiceConstants.SERVICEPROVIDER, serviceProvider);
    } catch (Exception e) {
        logger.warn("Failed to initialize RemoteServiceProvider (Degraded mode): " + e.getMessage(), e);
        sc.setAttribute(WFCIServiceConstants.SERVICEPROVIDER, null);
    }
    return serviceProvider;
}
```
