我覺得亂的原因是：

1. **三個角色的層級不一致**
    
    - RemoteInvocation 是「資料」
        
    - RmiInvocationHandler 是「介面/服務」
        
    - RmiInvocationWrapper_Stub 是「實作物件」
        
2. 描述時混用了「本質、職責、比喻、方法」，讀者容易失焦。
    

如果是要放在技術分享或文件裡，我會改成先講「它們在 RMI 呼叫流程中的角色」，讓關係更清楚。

---

## ① RemoteInvocation：遠端呼叫請求

### 角色

封裝一次遠端方法呼叫所需的所有資訊。

### 內容

當 Client 呼叫遠端服務時，Spring 會將下列資訊封裝成 `RemoteInvocation`：

- Method Name（方法名稱）
    
- Parameter Types（參數型別）
    
- Arguments（參數值）
    

### 特性

- Serializable
    
- 本質上是一個 DTO
    
- 不負責執行任何邏輯
    

### 比喻

就像一張派工單，記錄：

> 「要呼叫哪個方法、帶哪些參數」

---

## ② RmiInvocationWrapper_Stub：客戶端代理物件

### 角色

負責將 RemoteInvocation 傳送到遠端伺服器。

### 工作內容

收到：

```java
userService.queryUser("A123");
```

後，會：

1. 建立 RemoteInvocation
    
2. 將物件序列化
    
3. 透過 RMI 網路通訊傳送到 Server
    

### 特性

- Client 端產生的 Stub
    
- 實作 `RmiInvocationHandler`
    
- 隱藏底層 Socket 與 RMI 通訊細節
    

### 比喻

就像一支遙控器。

使用者按下按鈕時：

> 遙控器不負責執行動作，只負責把指令送到遠端設備。

---

## ③ RmiInvocationHandler：伺服器端呼叫入口

### 角色

接收來自 Client 的 RemoteInvocation，並轉交給真正的 Service 執行。

### 核心方法

```java
Object invoke(RemoteInvocation invocation)
```

### 工作內容

收到 RemoteInvocation 後：

1. 解析方法名稱
    
2. 解析參數資訊
    
3. 使用 Reflection 呼叫目標物件
    
4. 取得執行結果
    
5. 回傳結果給 Client
    

### 特性

- Server 端公開的 RMI Remote Interface
    
- 是 Client 與真正 Service 之間的橋樑
    

### 比喻

就像公司櫃台。

櫃台收到派工單後：

> 不自己處理工作，而是找到正確的工程師執行，再把結果回覆給客戶。

---

## 三者關係圖

```text
Client
  │
  │ userService.queryUser("A123")
  ▼
RmiInvocationWrapper_Stub
  │
  │ 建立
  ▼
RemoteInvocation
  │
  │ RMI 傳輸
  ▼
RmiInvocationHandler
  │
  │ Reflection
  ▼
Real Service
  │
  ▼
Result
```

一句話總結：

|元件|職責|
|---|---|
|RemoteInvocation|封裝呼叫資訊|
|RmiInvocationWrapper_Stub|傳送呼叫請求|
|RmiInvocationHandler|接收並執行呼叫|

這樣的結構比較符合技術文件閱讀習慣：**先講角色，再講職責，最後看整體流程**，而不是在每個段落裡同時混雜本質、比喻與方法。對晨會分享或架構文件來說，可讀性會高很多。

專案利用Spring 的 
RmiProxyFactoryBean 進行 RMI 遠端呼叫，其底層核心就是 JDK Dynamic Proxy
1. 初始化&註冊: PASEConfig.java
它有 @Configuration
會透過 @Configuration 建立並註冊 RemoteServiceProvider

委派給  spring 封裝的 RMI Stub，將要求送 RMIServer 進行處理。
RmiProxyFactoryBean 本身就是一個 MethodInterpretor
afterPropertiesSet()，會初始化一個 proxyFactory，並將  bean 本身加入代理配置。
最後用 JDK Dynamic Proxy 產生代理實例: 
this.serviceProxy = proxyFactory.getProxy(getBeanClassLoader())
故，呼叫代理方法時，會自動進入 RmiClientInterpretor.invoke() 進行攔截。
