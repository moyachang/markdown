# <font color="green">WFCIService 重構後 Spring Remoting 相容處理</font>
【分享原因】 
> WFCIService 因重構版本升級，Spring Boot 由 2.5.12 升級至 3.5.6 ，其中 Spring Remoting (RMI) 因為安全性問題，已被移除。
而 WFCIService 的 API 又大量的依靠 Spring Remoting (RMI) 技術，且因 PASE 目前無法更新 Spring boot 版本，因此 WFCIService 需要處理相容問題，分享處理過程學到的內容。

---

## 一、==何謂 RMI (Remote Method Invocation)?==
**遠端方法呼叫 (Remote Method Invocation)** 是 Java 原生的分散式通訊 API。
- **用途**：允許在某一台 Java 虛擬機 (JVM) 上運行的程式，直接呼叫另一台遠端伺服器（或不同網路位置）上的 Java 物件方法。
- **優點**：讓開發者在撰寫分散式應用程式時，不需要處理複雜的底層網路通訊細節。 


### 1. ==Spring Remoting RMI 呼叫流程==
Client 呼叫遠端服務時不能直接把像這樣的程式 userService.queryUser("A123")，送到 Server 執行。
必須經歷：**封裝呼叫資訊 -> 序列化傳送 -> Server 執行 -> 回傳結果**。

Spring 框架將傳統 RMI 的 `Stub` (客戶端代理) 與 `Skeleton` (伺服器端骨架) 進行了封裝，核心涉及三個關鍵類別：

他們在架構中的傳遞流程如下：

[![](http://hedgedoc.flowring.com/uploads/f0a160a8-a4b0-4641-9216-21401dfb9cb1.png)](http://hedgedoc.flowring.com/uploads/f0a160a8-a4b0-4641-9216-21401dfb9cb1.png "點我放大")


### 2. ==三個核心元件與職責==
- 
    | 元件名稱 | 職責與角色 | 實務運作說明 |
    | --- | --- | --- |
    | **RemoteInvocation** | **封裝呼叫資訊** | 本質上是一個 `Serializable` 的 DTO（派工單），記錄了 Method Name、Parameter Types 與 Arguments。不負責執行任何邏輯。 |
    | **RmiInvocationWrapper\_Stub** | **傳送呼叫請求** | 實作了 `RmiInvocationHandler` 介面，存在於 Client 端的代理物件（遙控器），負責將 RemoteInvocation 序列化後透過網路送到 Server。 |
    | **RmiInvocationHandler** | **接收並執行呼叫** | Server 端公開的 RMI 遠端介面（公司櫃台）。收到 RemoteInvocation 後，解析其方法與參數，還原並執行真正的 Service，再將結果回傳。 |

    > **💡 補充名詞：Stub (虛設常式)** 在台灣開發術語中，Stub 常指測試時寫死回傳值（Hard-code）的假功能（Mock 兄弟類）。但在 RMI 網路架構中，Stub 專指「客戶端代理存根」，負責處理底層 Socket 與通訊，讓本機程式誤以為遠端服務就在本機一樣。

## 二、==RMI 在 WFCIService 的工作情境==
- 1. **分散式通訊的橋樑**
WFCIService本身並不直接執行工作流引擎與底層業務的核心代碼，而是利用 RMI Client 連線到遠端的 PASE RMI Server。
    
- 2. **透過 Spring 動態代理查找 (Lookup) 遠端服務**
在 PASEConfig.java 中，原先會利用 Spring RmiProxyFactoryBean 對接 RMI 註冊表 (Port 1099)，動態查找並包裝以下 PASE 核心遠端服務接口:
    •  `DBService`：資料庫與內部數據存取服務。
    •  `FileService`：遠端文件與附件上傳/下載管理。
    •  `FlowService`：核心流程、表單以及 BPM 工作流引擎運作控制。
    •  `OrgService`：組織架構、角色、部門與人員權限等安全控制。
    •  `PDEService` / `PDEServerserService`：流程定義與部署調試相關服務。

- 3. **初始化 WFCI 執行實體**
建立服務提供者後，注入 WFCIPool 產生 WFCI (Workflow Client Interface)物件，供 Controller / Service 層使用。

        ```java=
        @Bean
        public RemoteServiceProvider serviceProvider() {
            return new RmiRemoteServiceProvider(sysConfig().getPaseIP(), sysConfig().getPasePort());
        }

        @Bean
        public WFCI wfci() {
            WFCIPool.destroyPool();
            WFCIPool.initPool(serviceProvider()); // 將 RmiRemoteServiceProvider 注入
            return WFCIPool.getWFCI();
        }
        ```

### **【技術考古: 為什麼 Ant 打包腳本有 `*_Stub.class` 但實體目錄找不到？】**
- build_dev.xml 裡的打包描述中，可以看到 *_stub.class，但實際去找是找不到的。
    ```xml=
    <jar jarfile="adm.jar">
       <fileset dir="${buildDir}">
         <include name="pase/admin/*.class" />
        <include name="pase/agenda/FlowClient_Stub.class" />
        <include name="pase/server/ServerServiceImpl_Stub.class" />
        ..........略
        </fileset>
    </jar>
    ``` 
- **以前**：
Java 5 之前，必須使用 JDK 的 `rmic` (RMI Compiler) 工具對實作類別進行二次編譯，手動產出這些 Stub 實體檔案，否則 Client 無法連線。
Client 必須透過編譯出來的 FlowServiceImpl_Stub 作為網路代理。
當 Client 呼叫 FlowService 介面的方法時，實際上是呼叫這個 Stub，由它進行序列化、透過 Socket 傳送到 Server、並接收回傳值。
在 PASE 專案的 Makefile（第 481 行）中，可以清楚看到這段歷史痕跡：
    ```bash=
    # rmic -classpath . -d . pase.server.FlowServiceImpl
    ```
- **現在（後期）**：
    1.  Java 5+ 支援動態 Stub (Dynamic Proxy Stub)。
    2.  專案導入 Spring Remoting，`RmiServiceExporter` 會在執行期（Runtime）自動透過 Java 動態代理產生通訊代理。
- **結論**：打包腳本中的 `*_Stub.class` 是沿用舊版本的**歷史遺留設定**，目前 Ant 執行時找不到會自動忽略 (Ignore)，實質上通訊早已交由 Spring 執行期動態代理處理。

## 三、==重構升版後的相容問題與方案選擇==
### 1. 為什麼不能直接使用 Spring Remoting ?
- 在 **Spring Boot 2.4 / Spring Framework 5.3（宣告棄用）**。
- 在 **Spring Boot 3.0 / Spring Framework 6.0（正式移除）**。
- Spring Boot 3.0 / Spring Framework 6.0 徹底移除了該功能，主要原因為：
    1. **Java 反序列化漏洞 (Security Vulnerability)**：
    RMI 機制的基礎是 Java 原生的序列化。攻擊者只要偽造一個惡意的 `RemoteInvocation` 包裹發送給伺服器，伺服器在反序列化解包的瞬間就會執行惡意程式碼，這在當今的資安環境下是非常致命的漏洞。
    2. **防火牆不友善**：
    RMI 使用隨機或特定的 TCP 連接埠通訊，企業防火牆極難管理。現代微服務更偏向使用標準的 HTTP (80/443) 連接埠。
    3. **跨語言能力差**：
    `RemoteInvocation` 和 Stub 架構完全綁死在 Java 語言上。如果前端想用 Node.js，或者其他微服務想用 Python/Go，RMI 完全無法溝通。

### 2. 方案評估
- WFCIService 採用 **Strangler Pattern (絞殺者模式)** 進行漸進式遷移，舊的 API 必須繼續運行。
    *   ❌ **方案 A：升級 PASE Server** -\> 現階段客觀因素無法動彈。
    *   ❌ **方案 B：改用純 JDK RMI** -\> 協定不相容，因為 PASE Server 依然是 Spring Remoting 封裝。
    *   ❌ **方案 C：自行重寫所有 Remoting 類別** -\> 工程量巨大，不切實際。
    *   ✅ **方案 D：引入舊版隱含依賴 + 自建相容代理層** -\> **最終採用方案**。

## 四、==相容層運作架構與原理==
- ### 方案選擇（方案 D：引入舊版隱含依賴 + 自建相容代理層）
    - 為了實現上述方案，設計了新的通訊相容層。
以下為 WFCIService (Spring Boot 3.x) 與 PASE Server (Spring 2.x) 之間的 RMI 通訊原理架構圖：
![WFCIService 相容層運作原理](http://localhost:3000/uploads/a7d307bd-934c-46a5-a0f0-f13df77bd89e.png)

- ### 💡 架構關鍵解析：
    - 這一次相容處理的核心邏輯，本質上就是在不變動 PASE 端（Provider）的前提下，由 WFCIService 端（Consumer）自主實作一條『向後相容的 RPC 通訊管道』。
我們可以把整個運行流程拆解為三個關鍵階段：

        - 1\. **容器級別的依賴解耦（PASEConfig 到自建相容層）**
由於 Spring Boot 3 徹底移除了 RmiProxyFactoryBean 這些傳統的 Remoting 類別，我們不再依賴框架自帶的組態。而是直接在 PASEConfig 中，將遠端連線生命週期委託給自定義的 CompatibleRmiRemoteServiceProvider。這讓我們能自主控制初始化時機，並為後續的 @Lazy 延遲載入與容錯降級（try-catch）提供切入點。

        - 2\. **執行期二進位序列化協定的對接（Registry Lookup 階段）**
當我們調用 JDK 原生的 LocateRegistry 去向遠端進行 lookup 時，舊版 PASE Server 回應的 Stub 封裝格式，在新版 JDK/Spring 的 ClassLoader 中是無法被反序列化的。
因此，我們手動將 spring-remoting-2.0.5.jar 引入新專案的 classpath。它的核心職責，就是在 lookup 的當下，提供執行期所需的 RmiInvocationWrapper_Stub 等傳輸類別，確保新舊兩端的二進位序列化協定（Serialization Protocol）完全一致。

        - 3\. **宣告式介面到遠端調用器的動態綁定（建立 JDK Dynamic Proxy）**
拿到遠端的 RmiInvocationHandler 入口後，我們利用 JDK Dynamic Proxy (動態代理) 在記憶體中動態產生業務介面（如 DBService）的代理實例。
當業務層（Controller/Service）調用該介面的任意方法時，代理會攔截請求，將『方法名稱、參數型別、實體參數值』手動封裝進舊版規格的 RemoteInvocation 物件中，最後經由 handler.invoke() 發送。對遠端的 PASE Server 而言，這個封裝格式與它預期接收的完全相同，因而能順利透過 Reflection 執行並回傳結果。

    - 總結報告： 
    > 藉由『外掛舊版序列化類別』加上『手動封裝 RPC(Remote Procedure Call) 請求』，在完全不改動遠端 PASE 的情況下，解決了 Spring 6 跨世代的 Remoting 斷層問題。」

## 五、==核心實作調整==

直接使用 spring-remoting-2.0.5.jar 中原始的 RmiInvocationHandler、RemoteInvocation、RmiInvocationWrapper_Stub 等類別， 確保 Client 和 Server 使用的是同一套序列化協定。
    
#### (一) 將 `spring-remoting-2.0.5.jar` 加入 Maven
- 父 pom.xml 宣告版本
    ```xml=
        <!-- 檔案：pom.xml (父 pom) -->
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.springframework</groupId>
                <artifactId>spring-remoting</artifactId>
                <version>2.0.5</version>
            </dependency>
        </dependencies>
    </dependencyManagement>
    ```    
- common/pom.xml - 加入依賴
    ```xml=
    <!-- 檔案：common/pom.xml -->
    <dependencies>
        <dependency>
            <groupId>org.springframework</groupId>
            <artifactId>spring-remoting</artifactId>
        </dependency>
    </dependencies>
    ```
    :::success
    **spring-remoting-2.0.5.jar 提供了 PASE Server RMI 通訊所需的關鍵類別：**

    - RmiInvocationWrapper_Stub（RMI 反序列化必需）
    - RmiInvocationHandler（遠端呼叫介面）
    - RemoteInvocation（封裝方法呼叫資訊）
    - RemoteAccessException / RemoteLookupFailureException（例外類別）    
    :::

- 安裝到 Local Maven Repository    
    如果 Maven Central 沒有 spring-remoting:2.0.5，需要手動安裝：
    ```bash=
    mvn install:install-file ^
      -Dfile=path/to/spring-remoting-2.0.5.jar ^
      -DgroupId=org.springframework ^
      -DartifactId=spring-remoting ^
      -Dversion=2.0.5 ^
      -Dpackaging=jar    
    ```
#### (二) 建立 RemoteServiceProvider 介面
- 這是一個自定義介面，定義了遠端服務查找的標準：
說明： 這個介面取代了舊版專案中直接使用的 RemoteServiceProvider（原本在 PASE 的 jar 中）， 讓 WFCIService 可以自行控制服務查找邏輯。    
    ```java=
        package com.flowring.remoting;

    public interface RemoteServiceProvider {
        /**
         * 根據服務名稱取得遠端服務代理
         * @param serviceName 服務名稱（如 "DBService"）
         * @return 遠端服務的動態代理物件
         */
        Object getService(String serviceName) throws Exception;

        /**
         * 取得 RMI Registry URL
         */
        String getRmiRegistryUrl();
    }
    ```
    
#### (三) 建立 CompatibleRmiRemoteServiceProvider 實作
- 核心邏輯：
    下述 code 第 5 點。
    <font color="red">這個 Dynamic Proxy 的 handler，實際上就是手動在做舊版 RmiProxyFactoryBean 背景做掉的封裝與 RMI 反序列化呼叫。</font>
    ```java=
    @Override
    public Object getService(String serviceName) throws RemoteLookupFailureException {
        // 1. 查詢快取
        Object cached = serviceCache.get(serviceName);
        if (cached != null) return cached;

        // 2. 根據服務名稱找到對應的 Java 介面 (例如 "DBService" → pe.pase.DBService)
        Class<?> serviceInterface = Class.forName(SERVICE_INTERFACE_MAP.get(serviceName));

        // 3. 從 RMI Registry 查找（此處需要 spring-remoting-2.0.5.jar 中的 Stub 類別做反序列化）
        Remote remoteObj = (Remote) registry.lookup(serviceName);

        // 4. 確認是 RmiInvocationHandler（Spring Remoting 協定）
        RmiInvocationHandler handler = (RmiInvocationHandler) remoteObj;

        // 5. 建立 JDK Dynamic Proxy，將方法呼叫轉為 RemoteInvocation
        Object proxy = Proxy.newProxyInstance(
            serviceInterface.getClassLoader(),
            new Class<?>[] { serviceInterface },
            (proxyObj, method, args) -> {
                RemoteInvocation invocation = new RemoteInvocation(
                    method.getName(), method.getParameterTypes(), args);
                return handler.invoke(invocation);
            }
        );

        // 6. 快取並回傳
        serviceCache.put(serviceName, proxy);
        return proxy;
    }    
    ```
    服務介面對應表
    
    ```java=
    SERVICE_INTERFACE_MAP.put("DBService",        "pe.pase.DBService");
    SERVICE_INTERFACE_MAP.put("FileService",      "pe.pase.FileService");
    SERVICE_INTERFACE_MAP.put("FlowService",      "pe.pase.FlowService");
    SERVICE_INTERFACE_MAP.put("OrgService",       "pe.pase.OrgService");
    SERVICE_INTERFACE_MAP.put("PDEService",       "pe.pase.PDEService");
    SERVICE_INTERFACE_MAP.put("SecurityService",  "pe.pase.SecurityService");
    SERVICE_INTERFACE_MAP.put("UtilityService",   "pe.pase.UtilityService");
    SERVICE_INTERFACE_MAP.put("ReportService",    "pe.pase.ReportService");
    SERVICE_INTERFACE_MAP.put("IAService",        "com.flowring.ita.IAService");
    SERVICE_INTERFACE_MAP.put("ActivityService",  "pe.pase.ActivityService");
    SERVICE_INTERFACE_MAP.put("PDEServerService", "pe.pde.PDEServerService");
    SERVICE_INTERFACE_MAP.put("CompressService",  "pe.pase.CompressService");
    SERVICE_INTERFACE_MAP.put("DashboardService", "pe.pase.DashboardService");
    SERVICE_INTERFACE_MAP.put("LogService",       "pe.pase.LogService");
    ```
    
#### (四) 修改 PASEConfig 使用新的相容層
- 
    > **關鍵改動：**
    > - 加入 `@Lazy` 避免啟動時因 PASE Server 未啟動而導致整個應用啟動失敗
    > - 加入 try-catch 讓連線失敗時應用仍能啟動（降級運行）
    > - 改用 `CompatibleRmiRemoteServiceProvider` 取代舊的 `RmiRemoteServiceProvider`

- **修改前（Spring 5 時代）：**
    ```java=
    @Bean
    public RemoteServiceProvider serviceProvider() {
        // 舊版直接用 PASE jar 中的 RmiRemoteServiceProvider
        serviceProvider = new RmiRemoteServiceProvider(sysConfig().getPaseIP(), sysConfig().getPasePort());
        sc.setAttribute(WFCIServiceConstants.SERVICEPROVIDER, serviceProvider);
        return serviceProvider;
    }
    ```

- **修改後（Spring 6 相容層）：**
    ```java=
    @Bean
    @Lazy
    public RemoteServiceProvider serviceProvider() {
        RemoteServiceProvider serviceProvider = null;
        try {
            logger.info("Attempting to initialize CompatibleRmiRemoteServiceProvider...");
            serviceProvider = new CompatibleRmiRemoteServiceProvider(
                sysConfig().getPaseIP(), sysConfig().getPasePort());
            sc.setAttribute(WFCIServiceConstants.SERVICEPROVIDER, serviceProvider);
            logger.info("RemoteServiceProvider initialized successfully");
        } catch (Exception e) {
            logger.warn("Failed to initialize RemoteServiceProvider: " + e.getMessage(), e);
            logger.warn("Continuing without RemoteServiceProvider.");
            sc.setAttribute(WFCIServiceConstants.SERVICEPROVIDER, null);
        }
        return serviceProvider;
    }

    @Bean
    @Lazy
    public WFCI wfci() {
        try {
            WFCIPool.destroyPool();
            RemoteServiceProvider provider = serviceProvider();
            if (provider != null) {
                WFCIPool.initPool(provider);
            }
            return WFCIPool.getWFCI();
        } catch (Exception e) {
            logger.error("Error initializing WFCI: " + e.getMessage(), e);
            return null;
        }
    }
    ```
    
#### (五) 異動清單
- 新增
    - RemoteServiceProvider: 自定義遠端服務提供者介面。
   - CompatibleRmiRemoteServiceProvider: 相容層實作（使用 spring-remoting-2.0.5.jar）。
- 修改
    - pom.xml（父）: dependencyManagement 加入 spring-remoting:2.0.5
    - common/pom.xml: 	dependencies 加入 spring-remoting
    - PASEConfig.java: 改用 CompatibleRmiRemoteServiceProvider，加入 @Lazy 和 try-catch
- 依賴 jar
    - spring-remoting-2.0.5.jar: 舊版 Spring Remoting jar，提供 RmiInvocationWrapper_Stub 等序列化類別。
