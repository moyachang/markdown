「這張圖清楚呈現了 **Spring Remoting over RMI 的底層呼叫流程，以及三個核心元件之間的關係**。當我們在 Client 端呼叫一個遠端服務時，底層其實經歷了從左到右的封裝與傳輸。

我們首先看左半邊**客戶端 (Client)** 的四大步驟：

當我們在 **步驟 1** 呼叫業務方法，例如 `userService.queryUser("A123")` 時，**步驟 2** 的 **JDK Dynamic Proxy** 會立刻攔截這個介面呼叫，並把它委派給 **步驟 3** 的 **RMI Stub**（也就是 `RmiInvocationWrapper_Stub`）。

這個 Stub 的核心工作有三個：第一，把方法名稱、參數型別、參數值，封裝成 **步驟 4** 的 **`RemoteInvocation`** 這張遠端呼叫 DTO；第二，將物件序列化；第三，透過 RMI 網路協定傳送出去。這整個過程完美隱藏了底層複雜的 Socket 網路細節。

接下來看右半邊**伺服器端 (Server)** 的處理：

資料跨網路傳輸過去後，會由 **步驟 5** 的 Server 端入口 —— **`RmiInvocationHandler`** 負責接收。進入 **步驟 6**，Server 會去解析這張 `RemoteInvocation` DTO 裡面的呼叫資訊，拆解出方法名稱與參數。

確認資訊後，在 **步驟 7** 透過 **Java Reflection（反射）** 機制去對號入座，找到目標物件，並在 **步驟 8** 驅動 **實際的目標服務** 去執行業務邏輯。執行完畢後，結果會再經過反序列化，一路回傳給 Client 端。

**最後看下方方框，簡單總結這三者的重點關係：**

`RemoteInvocation` 就是純資料傳輸的 DTO；Client 端的 Stub 負責建立並傳送呼叫資訊；Server 端的 Handler 則負責接收並執行。兩端就是透過這樣的 RMI 機制，默默完成了這一次的遠端方法調用。

第二張圖-相容
   - 這一次相容處理的核心邏輯，本質上就是在不變動 PASE 端（Provider）的前提下，由 WFCIService 端（Consumer）自主實作一條『向後相容的 RPC 通訊管道』。
我們可以把整個運行流程拆解為三個關鍵階段：

        - 1\. **容器級別的依賴解耦（PASEConfig 到自建相容層）**
由於 Spring Boot 3 徹底移除了 RmiProxyFactoryBean 這些傳統的 Remoting 類別，我們不再依賴框架自帶的組態。而是直接在 PASEConfig 中，將遠端連線生命週期委託給自定義的 CompatibleRmiRemoteServiceProvider。這讓我們能自主控制初始化時機，並為後續的 @Lazy 延遲載入與容錯降級（try-catch）提供切入點。

        - 2\. **執行期二進位序列化協定的對接（Registry Lookup 階段）**
當我們調用 JDK 原生的 LocateRegistry 去向遠端進行 lookup 時，舊版 PASE Server 回應的 Stub 封裝格式，在新版 JDK/Spring 的 ClassLoader 中是無法被反序列化的。
因此，我們手動將 spring-remoting-2.0.5.jar 引入新專案的 classpath。它的核心職責，就是在 lookup 的當下，提供執行期所需的 RmiInvocationWrapper_Stub 等傳輸類別，確保新舊兩端的二進位序列化協定（Serialization Protocol）完全一致。

        - 3\. **宣告式介面到遠端調用器的動態綁定（建立 JDK Dynamic Proxy）**
拿到遠端的 RmiInvocationHandler 入口後，我們利用 JDK Dynamic Proxy (動態代理) 在記憶體中動態產生業務介面（如 DBService）的代理實例。
當業務層（Controller/Service）調用該介面的任意方法時，代理會攔截請求，將『方法名稱、參數型別、實體參數值』手動封裝進舊版規格的 RemoteInvocation 物件中，最後經由 handler.invoke() 發送。對遠端的 PASE Server 而言，這個封裝格式與它預期接收的完全相同，因而能順利透過 Reflection 執行並回傳結果。

    - 總結報告： 
    > 藉由『外掛舊版序列化類別』加上『手動封裝 RPC(Remote Procedure Call) 請求』，在完全不改動遠端 PASE 的情況下，解決了 Spring 6 跨世代的 Remoting 斷層問題。」
