# WFCIService 的 RMI 問題

## ==何謂 RMI ?==
RMI (Remote Method Invocation) 遠端方法呼叫
這是 Java 程式語言中的一種 API。
- **用途**：允許在某一台 Java 虛擬機 (JVM) 上運行的程式，直接呼叫另一台遠端伺服器（或不同網路位置）上的 Java 物件方法。
- **優點**：讓開發者在撰寫分散式應用程式時，不需要處理複雜的底層網路通訊細節。 
## ==RMI 在 WFCIService 的用途為何?==
主要的服務，就是透過 PASE  所提供的功能查詢資料後並回傳。再由 WFCIService 自行處理資料邏輯及要回傳到前端的內容。



## ==為何無法直接使用 Spring Remoting ?==
- 在 **Spring Boot 2.4 / Spring Framework 5.3（宣告棄用）**。
- 在 **Spring Boot 3.0 / Spring Framework 6.0（正式移除）**。
	隨著 Spring 邁入 6.0 世代（JDK 17 與 Jakarta EE 9+ 基準線線），官方為了移除老舊、不安全（容易引發 Java 反序列化漏洞）且不再符合現代微服務架構的技術，**正式將這些 Remoting 相關程式碼從核心框架中完全剔除**。
- 替代方案
	* **RESTful APIs**：使用 `WebClient` 或 Spring Boot 3.x 引入的全新 `RestClient` / `@HttpExchange` HTTP 介面客戶端。	
	* **gRPC / Protocol Buffers**：用於高效能、強型別的微服務間 RPC 通訊。	
	* **Spring Cloud OpenFeign**：若習慣宣告式的遠端呼叫介面，Feign 是目前生態系中非常主流的選擇。
而 WFCIService 因為重構的關係，Spring Boot 的版本已升到 3.5.6。
因為上述原因，導致 Spring Remoting ，無法直接使用。
專案雖然版本升級，但因為程式無法全面重構，勢必會需要新舊程式共存，舊的程式仍需要依賴此技術查詢資料。

因此，為了讓 **Spring Boot 3.x (Spring 6)** client 能與舊版 **Spring remoting server** 溝通，開始了一些研究。

## ==方案選擇==

| 方案 | 說明 | 是否可行 |
|------|------|---------|
| ❌ 升級 PASE Server | 改用 Spring 6 或 REST API | 現階段無法升級 |
| ❌ 純 JDK RMI | 直接用 `java.rmi` 連線 | 協定不相容，Server 用的是 Spring Remoting 封裝 |
| ❌ 自行完整實作 Spring Remoting | 重寫所有 remoting 類別 | 工程量太大，不實際 |
| ✅ **引入舊版 jar + 自建相容層** | 引入 `spring-remoting-2.0.5.jar` 提供序列化類別，自建 Provider 串接 | **採用方案** |

## ==實作==

- Spring Boot 3.x / Spring Framework 6.x 已移除 `spring-remoting` 模組

- 舊的 PASE Server 使用 Spring remoting 協定（`RmiInvocationWrapper`）

- 無法直接使用 JDK 原生 RMI 連線（協定不相容）
